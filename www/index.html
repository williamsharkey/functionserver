The Vision: Your Computer in the Cloud

Every user gets their own sandboxed environment. Install apps, save files, customize your desktop. It's your machine, running in the browser.

An Observation

What happens when apps are artifacts in a shared VM?

Live patching exists. Browser automation exists. Man-in-the-middle injection exists. So what's different here?

In FunctionServer, the apps are the JavaScript artifacts. They're not compiled binaries running behind an API. They're not isolated processes that need IPC to communicate. They're functions and objects living in the same VM that an AI agent can access via MCP.

This creates something unexpected: the AI doesn't automate the OS from outside. It inhabits it.

When Claude inspects a menu position, it's calling getBoundingClientRect() on the same DOM element the user sees. When it patches a function, that function is the real running code. When it injects a WebGL shader as a desktop background, the user watches it appear in real-time.

The line between "user" and "developer" and "operating system" starts to blur. The OS becomes less like a fixed container and more like a collection of morphing artifacts—apps that can be inspected, modified, and extended by anything with access to the VM.

We don't know yet what this enables. It's early. But there's something interesting in the topology of it—a cloud OS where the apps, the AI, and the debugging tools all share the same address space.

One thing becomes clear quickly: you can't version control a system that's constantly being reshaped. So you don't. Each app becomes its own repo. What you version is the protocol—the API, the conventions for how apps declare their inputs and outputs, how they register file types, how they communicate. The OS defines the rules. The user shapes whatever they want within them.

This is an experiment in leaving the architecture open.

App Architecture

A component-based system for building real applications

📄 Project Structure

Apps live in folders with multiple files. Layouts define screens, UI components are reusable, and the main .js file ties it together.

my-app/
├── main.js           # App entry point
├── screens/
│   ├── home.layout   # Screen layouts
│   └── settings.layout
├── components/
│   ├── button.ui     # Reusable UI
│   └── card.ui
└── app.json          # Metadata

📐 Designer Tool

Visual constraints-based layout editor. Drag components, set constraints (pin to edges, center, stack), preview on multiple sizes.

// Constraints example
{
  "type": "button",
  "text": "Submit",
  "constraints": {
    "bottom": 20,
    "centerX": true,
    "width": "80%"
  }
}

🔧 Craft Tool

Build reusable UI components. Define props, styles, and behavior. Components compile to efficient JavaScript.

// Component definition
ALGO.component('MyButton', {
  props: ['text', 'onClick'],
  render: (props) => `
    <button onclick="${props.onClick}">
      ${props.text}
    </button>
  `
});

📦 System Apps

Pre-installed apps managed by admins. Auto-update when the server updates. Users can fork to customize, or uninstall to hide.

// System apps loaded from /core/apps/
systemApps = [
  { id: 'shade-station', ... },
  { id: 'ticket-manager', ... },
  { id: 'photobooth', ... }
];
